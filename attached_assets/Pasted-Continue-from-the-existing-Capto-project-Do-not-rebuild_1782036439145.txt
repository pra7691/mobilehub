Continue from the existing Capto project. Do not rebuild authentication, tasks, capture, uploads, submissions, wallet, payouts, language support, admin layout, or mobile navigation.

Build the referral system with numeric six-digit referral codes and wallet reward credit after the referred user completes their first approved task.

Important rules:

* Referral codes must be numeric only.
* Every referral code must contain exactly 6 digits.
* Generate referral codes randomly.
* Codes must be unique.
* Example: `482917`
* Do not use letters, prefixes, or alphanumeric codes.
* Referral reward amount is configurable by admin.
* Reward is credited to the referrer’s wallet only after the referred user completes their first approved task.
* Do not credit reward merely when referred user registers or logs in.
* Do not credit reward multiple times for the same referred user.

## 1. Referral Code Generation

Update User model:

* referralCode: string
* Exactly 6 numeric digits
* Unique
* Generated automatically when a user account is created
* Existing users without a code must receive one through migration/backfill logic

Generation rules:

* Generate random number between 100000 and 999999.
* Check uniqueness before saving.
* Retry safely on collision.
* Add a unique database constraint.
* Never regenerate an existing user’s referral code unless explicitly required by admin-only repair logic.

## 3. Referral Reward Rules

Add referral settings in Admin Settings:

Referral Settings:

* Enable/disable referral program
* Referral reward amount in INR
* Default reward amount: ₹0 until admin configures it
* Reward trigger: first approved task completed by referred user
* Optional informational message shown in mobile referral screen

Reward flow:

1. User A shares referral code.
2. User B registers and applies User A’s referral code.
3. Referral record is created with status REGISTERED.
4. User B submits tasks.
5. When User B gets their first submission approved:

   * Check that referral program is enabled.
   * Check that referral reward has not already been paid for this referral.
   * Credit configured reward amount to User A wallet.
   * Create WalletTransaction.
   * Mark referral as QUALIFIED / REWARDED.
   * Store reward transaction ID and reward credited timestamp.
6. Do not give reward to User B in this version.
7. Do not pay the referrer again if User B completes more tasks.

Use an atomic database transaction for:

* Submission approval
* Referral qualification
* Wallet credit
* Referral reward transaction creation

Prevent duplicate rewards if:

* Admin double-clicks approval.
* Submission approval retries.
* Background jobs retry.
* Multiple submissions are approved at similar time.

## 8. Submission Approval Integration

Modify existing submission approval flow:

When admin approves a submission:

1. Approve submission as usual.
2. Credit the task payment to the submitting user as usual.
3. Check whether the submitting user is a referred user.
4. Check if their referral is REGISTERED and has no rewardWalletTransactionId.
5. Check whether this is the user’s first approved submission.
6. If yes and referral program is enabled and rewardAmount > 0:

   * Credit reward to referrer wallet.
   * Create referral reward WalletTransaction.
   * Update referral status to REWARDED.
   * Save firstQualifiedSubmissionId, reward amount, rewardedAt, and transaction ID.
7. If no, do nothing.

The referral reward must not interfere with normal task approval or payment credit.

## 9. Security and Edge Cases

* Referral code must remain numeric and exactly six digits.
* Use masked mobile numbers in referral history.
* Never expose unrelated user personal details.
* Do not allow self-referral.
* Do not allow multiple codes for one user.
* Do not allow referral relationship changes after application.
* Do not create reward if the referred user is disabled before first approved task, unless later approved after reactivation.
* Do not create reward if referral program is disabled at approval time.
* Keep audit logs for referral application, cancellation, qualification, and reward credit.
* Use transaction locking/idempotency to eliminate duplicate wallet credits.
