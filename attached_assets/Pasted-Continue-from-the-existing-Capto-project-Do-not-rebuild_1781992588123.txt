3. Backend APIs
   Create and document these APIs with Swagger.

Mobile-user authenticated APIs:

* POST /submissions/initiate

  * Creates a draft submission record from a local draft.
  * Validates user, task availability, limits, and task requirements.
  * Returns submission ID and upload targets/presigned URLs.

* POST /submissions/:id/upload-complete

  * Confirms upload completion.
  * Validates all required media were uploaded.
  * Changes submission status to UNDER_REVIEW.

* POST /submissions/:id/upload-failed

  * Marks failed upload with failure reason.

* GET /submissions/my

  * Returns current user submissions.
  * Supports status filter, pagination, and sorting.
  * Must include task snapshot and media preview details.

* GET /submissions/my/:id

  * Returns detailed submission information for the logged-in user only.

* DELETE /submissions/:id

  * Allow deletion only while status is DRAFT, UPLOADING, or UPLOAD_FAILED.
  * Do not allow deleting UNDER_REVIEW or later submissions.

Admin read APIs only for now:

* GET /admin/submissions
* GET /admin/submissions/:id

Admin endpoints should display submissions but must not approve/reject yet.

4. Upload Validation and Safety
   Before initiating upload, backend must validate:

* Task is active and available.
* User has not exceeded maxSubmissionsPerUser.
* Task has not exceeded maxTotalSubmissions.
* Video/audio duration is within task limits.
* Image count is within task limits.
* Collection type matches task requirement.
* Required media count exists.
* File type is acceptable.
* File-size limits are enforced through configurable server settings.
* User owns the local draft being submitted.

Use configurable maximum upload limits, with sensible defaults:

* Video: 500 MB
* Audio: 100 MB
* Image: 20 MB each

Do not trust metadata from mobile blindly. Store it, but mark it as client-captured metadata.

* Confirmation screen before upload showing:

  * task title
  * collection type
  * media count
  * duration/image count
  * expected payment amount
  * note: “Your submission will be reviewed before earnings are added.”

Upload flow:

1. User taps Upload Submission.
2. App calls initiate submission API.
3. App receives presigned upload URLs.
4. App uploads each local file directly to storage.
5. Show per-file and overall upload progress.
6. App calls upload-complete API.
7. Local draft is removed only after backend confirms UNDER_REVIEW status.
8. User sees “Submission uploaded successfully. It is now under review.”
9. Redirect to My Uploads → Under Review.

For failed uploads:

* Keep local draft and submitted upload state.
* Show retry button.
* Resume/retry only failed files when possible.
* Do not duplicate submission records accidentally.
* Use idempotency protection for initiate and completion requests.
