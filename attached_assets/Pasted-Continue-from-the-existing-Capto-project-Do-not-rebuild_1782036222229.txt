Continue from the existing Capto project. Do not rebuild authentication, task management, capture, uploads, submissions, wallet, payout flows, admin layout, or mobile navigation.

Implement full English and Hindi language support across the Expo mobile app, NestJS backend, PostgreSQL/Prisma schema, and Admin Dashboard.

Important product decision:

* English is the default language.
* Hindi must display as: `हिंदी`
* All mobile UI text must support English and Hindi.
* Admin-created content must support English and Hindi separately.
* Do not use automatic machine translation at runtime.
* Do not show blank Hindi content. When Hindi content is unavailable, fall back to English.

## 1. Mobile Language Selection Flow

Required flow for a new user:

Splash → Language Selection → Login → OTP → App

Language selection screen:

* Show app logo/name.
* Title: “Choose your language”
* Two options:

  * English
  * हिंदी
* English selected by default.
* User selects language and taps Continue.
* Save selection locally before login.
* Do not show language screen again after user has selected a language.
* On future launches:

  * If logged out: Splash → Login
  * If logged in: Splash → App
* User can change language later from Profile.

Use secure/local persistent storage already used in the project.

## 2. Mobile Profile Language Setting

Add a Profile menu item:

* Language

Language screen should show:

* English
* हिंदी
* Current selected language indicator
* Save/apply immediately when user taps a language

When user changes language:

* Update app UI immediately.
* Save locally.
* Sync selected language to backend user profile when logged in.
* Refresh task/category/subcategory/content data in the selected language.
* Keep user on the same screen where practical.

## 4. Backend User Language Preference

Add language preference to User model:

* preferredLanguage: `en` or `hi`
* Default: `en`

Add mobile authenticated API:

* GET /users/me
* PATCH /users/me/language

Example request:

```json
{
  "preferredLanguage": "hi"
}
```

Rules:

* Valid values only: `en`, `hi`
* Disabled users remain blocked as per existing disabled-account rules.
* API must return current preferred language in user profile response.
* Add Swagger docs and DTO validation.

## 7. Backend Content APIs

Update mobile-facing content APIs to support language parameter.

Examples:

* GET /categories?language=en
* GET /categories?language=hi
* GET /categories/:id/subcategories?language=hi
* GET /tasks?subcategoryId=...&language=hi
* GET /tasks/:id?language=hi
* GET /public/faq?language=hi
* GET /app/legal/privacy-policy?language=hi
* GET /app/legal/terms-and-conditions?language=hi
* GET /app/settings?language=hi
* GET /app/notices?language=hi

Rules:

* If Hindi content exists, return Hindi content.
* If Hindi content is missing, return English text as fallback.
* API response may include:

  * requestedLanguage
  * resolvedLanguage
  * fallbackUsed: true/false
* Do not return both full English and Hindi content in every mobile response unless needed.
* Admin APIs should return both languages for editing.

## 8. Prisma / Database Migration

Update Prisma schema safely.

Requirements:

* Add `preferredLanguage` to User.
* Add bilingual fields to content entities.
* Preserve existing English content by migrating current fields into English fields.
* Do not lose existing categories, subcategories, tasks, FAQ, legal content, support content, or notices.
* Existing single-language data should become English by default.
* Hindi fields can initially be null/empty.
* Add migration notes.

## 9. Fallback Rules

Implement these strict rules:

1. English selected:

* Always show English content.

2. Hindi selected:

* Show Hindi content when available.
* Otherwise show English content.
* Never show blank category title, task instruction, FAQ answer, legal page, or support text.

3. UI text:

* Hindi UI translations should always come from `hi.json`.
* Do not fall back to raw translation keys.

4. Legal pages:

* Show Hindi content when published Hindi content exists.
* Otherwise show English published content.
* Do not hide the legal page merely because Hindi is unavailable.

## 10. Safe Rollout

Do not break existing users.

For users who already exist:

* Default preferredLanguage to `en`.
* Existing login/session must continue to work.
* Existing admin-created content must remain visible in English.

For mobile:

* Existing users should see English unless they change language from Profile.
* New users choose language before login.
