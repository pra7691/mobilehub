Continue from the existing Capto project. Do not rebuild authentication, submissions, wallet credit, uploads, task management, capture flows, admin layout, or profile navigation.

Build a complete manual UPI payout system.

Important:

* There is no real payout provider yet.
* Do not integrate RazorpayX, Cashfree, PhonePe, Paytm, bank APIs, or any automatic UPI transfer service.
* Admin will manually send money through their own UPI/banking app, then mark the payout as paid in Capto.
* The system must track payout requests, wallet holds, payment status, and payout reference details correctly.

Goal:

1. User can save one UPI ID in Profile.

2. User can request withdrawal from their available wallet balance.

3. Admin can see payout requests and manually transfer money outside the system.

4. After manually paying, admin marks request as Paid and enters a payment reference/UTR/transaction ID.

5. User can see payout history and paid status.

5. Wallet Accounting Rules

All payout actions must use a database transaction and locking/idempotency protection.

When user creates a withdrawal request:

* Confirm wallet has sufficient available balance.
* Create PayoutRequest with PENDING status.
* Reduce availableBalance by requested amount.
* Increase pendingWithdrawalBalance by requested amount.
* Create WalletTransaction with PAYOUT_HOLD.
* Store walletHoldTransactionId on payout request.

When admin starts processing:

* Change PayoutRequest from PENDING to PROCESSING.
* Do not alter wallet balances again.
* Store processingStartedAt and processedByAdminId.

When admin marks payout as PAID:

* Only allow from PROCESSING status.
* Admin must enter payoutReferenceId, such as UTR, bank transaction ID, or UPI reference ID.
* Change status to PAID.
* Reduce pendingWithdrawalBalance by payout amount.
* Create WalletTransaction with PAYOUT_COMPLETED.
* Store paidAt, processedAt, processedByAdminId, payoutReferenceId, and walletCompletionTransactionId.
* Do not change availableBalance again because amount was already held.

When admin rejects payout:

* Allow from PENDING or PROCESSING.
* Rejection reason is required.
* Change status to REJECTED.
* Reduce pendingWithdrawalBalance by payout amount.
* Return payout amount to availableBalance.
* Create WalletTransaction with PAYOUT_REVERSED.
* Store rejectedAt, processedByAdminId, and rejectionReason.

When user cancels payout:

* Allow only from PENDING.
* Change status to CANCELLED.
* Reduce pendingWithdrawalBalance by payout amount.
* Return amount to availableBalance.
* Create WalletTransaction with PAYOUT_REVERSED.

Prevent:

* Double tap creating duplicate payout requests.
* Withdrawal above available balance.
* Duplicate processing/paid actions.
* Multiple wallet holds for the same request.
* Marking a payout paid twice.
* Deleting a UPI method linked to an active payout request.
* Editing UPI ID while payout status is PROCESSING.

* UPI verification is manual in this version.

* `mark-paid` does not trigger any payment API. It only records that admin has already paid manually.

9. Security and Privacy

* Store full UPI ID securely; encrypt at rest if current architecture supports encryption.
* Always store a masked UPI ID separately for safe display.
* Never include full UPI IDs in normal logs, mobile error logs, analytics, notifications, or list tables.
* Never log access tokens, refresh tokens, OTPs, or payment-sensitive values.
* Only admin users may verify UPI IDs or change payout status.
* Create audit logs for:

  * UPI ID add/update
  * UPI ID verification/rejection
  * Payout request
  * Payout cancellation
  * Start processing
  * Mark paid
  * Payout rejection
* Add idempotency protection to payout request creation and mark-paid action.
