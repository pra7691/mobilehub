Continue from the existing Capto project. Do not rebuild authentication, submissions, wallet credit, uploads, tasks, or existing profile screens.

Build a UPI payout request and admin payout-management module.

Goal:

* User can add and manage one UPI ID in Profile.
* User can request withdrawal of available wallet balance.
* Admin can review the request and mark it paid or rejected.
* Do not integrate automatic bank/UPI transfer API yet.
* Do not deduct or permanently reduce wallet balance until payout is marked paid by admin.

* Example helper text: `example@upi`

2. Mobile Wallet — Withdrawal Request
   On wallet/balance screen, add:

* Withdraw button

Rules:

* User can request withdrawal only if available balance is greater than or equal to configurable minimum withdrawal amount.
* Default minimum withdrawal amount: ₹100.
* User must have a saved UPI ID before requesting withdrawal.
* Show available balance.
* User enters withdrawal amount.
* Amount cannot exceed available balance.
* Show confirmation screen:

  * Withdrawal amount
  * Masked UPI ID
  * Processing note:
    “Your withdrawal request will be reviewed before payment is processed.”

After request:

* Create payout request.
* Move requested amount from available balance to pending withdrawal balance.
* Show request in payout history.
* Do not mark it paid yet.

5. Wallet Accounting Rules
   Use database transactions for every payout action.

When user creates payout request:

* Check available balance.
* Create PayoutRequest with PENDING status.
* Reduce available balance.
* Increase pending withdrawal balance.
* Create WalletTransaction with type PAYOUT_HOLD.

When admin marks payout as PAID:

* Reduce pending withdrawal balance.
* Create WalletTransaction with type PAYOUT_COMPLETED.
* Save payout reference ID.

When admin rejects payout:

* Reduce pending withdrawal balance.
* Return amount to available balance.
* Create WalletTransaction with type PAYOUT_REVERSED.
* Save rejection reason.

When user cancels pending request:

* Return amount to available balance.
* Create WalletTransaction with type PAYOUT_REVERSED.

Prevent:

* Duplicate payout requests caused by double tapping.
* Withdrawal beyond available balance.
* Multiple payout processing actions at the same time.
* Editing payment method while a payout request is PROCESSING.

8. Security

* Never expose full UPI IDs in public/mobile list APIs unless needed for the owner.
* Mask UPI IDs in logs and admin tables.
* Do not log UPI IDs in error logs.
* Add idempotency protection for withdrawal creation and payout completion.
* Restrict payout actions to admin users.
* Add audit log entries for UPI updates, payout requests, payout approval, payout rejection, and payout completion.
