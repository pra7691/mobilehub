Continue from the existing Capto project. Do not rebuild authentication, tasks, uploads, submissions, wallet, payouts, referral, language, or existing admin layout.

Fix country-code, phone-number storage, currency display, and UPI payout flow.

There are two main requirements:

1. India is currently the only supported country, with country code +91.
2. UPI ID verification must be removed completely.

## A. Phone Number and Country Code

Current behavior:

* Login screen visually has +91 hard-coded.
* But the stored user phone number may not include the country code consistently.
* The system must always store the complete normalized phone number.

Required behavior:

* India only for now.
* Country code is fixed to `+91`.
* User should enter only a 10-digit Indian mobile number.
* On login/request OTP/account creation:

  * Normalize number to E.164 format: `+91XXXXXXXXXX`
  * Store full number in database.
  * Use the same full number for OTP sessions, user uniqueness checks, login, referrals, admin users table, submissions, payouts, and audit logs.
* Do not let user manually edit/select country code in this version.
* Show +91 as fixed prefix in login UI.
* Validate:

  * exactly 10 digits after +91
  * no spaces, dashes, or duplicate +91
  * reject invalid Indian number format
* Existing users stored with only 10 digits must be migrated/backfilled safely to `+91XXXXXXXXXX`.
* Existing users already stored with +91 must not become `+91+91...`.
* Masked mobile display should use normalized number, for example:

  * `+91 98****4321`

Update:

* OTP request flow
* OTP verification flow
* user creation
* user lookup
* admin user lists/details
* referral flow
* payout flow
* mobile profile
* error logs
* any phone-number search/filter logic

## B. Currency and Amount Formatting

Current issue:

* Some screens show `$`.
* Some screens show ₹.
* Currency must be consistent.

Required behavior:

* Since only India / +91 is supported right now, the application currency must be INR everywhere.
* Display all user-facing monetary amounts using:

  * Symbol: `₹`
  * Currency: INR
  * Indian number formatting where practical, for example:

    * ₹40
    * ₹1,250
    * ₹12,500.50

Do not dynamically derive different currencies yet because country selection is not supported.

Create one centralized money-formatting utility used everywhere:

* Mobile app
* Admin dashboard
* Backend response formatting only where necessary

Never hard-code `$` in individual components.

Update all monetary areas:

* Task payout amount
* Task cards
* Task detail
* Submission expected payment
* Approved payment
* Wallet available balance
* Wallet pending earnings
* Wallet lifetime earnings
* Wallet statement
* Referral reward
* Payout request amount
* Payout history
* Admin task payment fields
* Admin submission approved amount
* Admin wallet transactions
* Admin payout screens
* Dashboard charts/cards
* Any notification or message mentioning money

Database rules:

* Existing currency fields should default to `INR`.
* Migrate incorrect default `USD` values to `INR` where they are Capto internal task/payment/payout records.
* Preserve numeric amounts correctly.
* Ensure all new tasks default to INR.
* Remove any UI currency selector for users.
* Admin task/payment forms should default to INR and not show USD unless multi-country support is intentionally added later.

Use a central formatter, for example:

* `Intl.NumberFormat('en-IN', { style: 'currency', currency: 'INR' })`
* For Hindi UI, still use ₹ and INR values, with Hindi labels around it.

## C. Remove UPI ID Verification Completely

Current issue:

* App shows “UPI ID verification pending.”
* User is blocked from requesting payout until admin verifies UPI ID.
* This is not required.

Required behavior:

* Remove UPI verification status from the user experience and payout workflow.
* When user adds a valid UPI ID, it becomes usable immediately.
* User can request payout immediately after saving UPI ID, provided they have sufficient available balance.
* Admin does not need to verify or approve UPI ID.
* Admin can still see the saved UPI ID in payout request details to pay manually.

Remove:

* Pending Verification status
* Verified status
* Rejected status
* Verify UPI button
* Reject UPI button
* UPI verification filters
* UPI verification APIs
* UPI verification-related admin actions
* User-facing message about verification pending
* Any payout blocking based on UPI verification

Simplify payment method model:

* Keep one active UPI ID per user.
* Fields:

  * id
  * userId
  * type: UPI
  * upiIdEncrypted or securely stored value
  * upiIdMasked
  * isDefault
  * createdAt
  * updatedAt

Migration:

* Existing UPI IDs with PENDING_VERIFICATION, VERIFIED, or REJECTED should become usable payment methods.
* Preserve the UPI ID itself.
* Remove old verification dependency safely.
* Do not delete payout requests that already exist.

## D. Simplified Manual Payout Flow

User:

1. Add UPI ID.
2. Open Wallet.
3. Tap Withdraw.
4. Enter amount.
5. Submit payout request.

Admin:

1. Open Payouts.
2. View payout request.
3. Reveal full UPI ID only through deliberate action.
4. Manually pay from external UPI/banking app.
5. Enter UTR/transaction/reference ID.
6. Mark payout as Paid.

Payout statuses remain:

* PENDING
* PROCESSING
* PAID
* REJECTED
* CANCELLED

Do not add automatic payout provider integration.
